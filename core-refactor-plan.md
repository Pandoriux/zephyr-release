# 🚀 Zephyr Release: Strategy & Lifecycle Architecture Refactor Guide

> **Note:** This is the general direction/flow, adapt it to your project/codes. This guide outlines the architectural philosophy and structural changes needed to support both `review` and `auto` modes seamlessly.

## 1. Configuration Schema Redesign (`config-options.md`)

To maximize Developer Experience (DX) and make your auto-generated documentation read chronologically, the root configuration must separate **Domain Rules** (what we do), **Execution Modes** (how we deliver it), and **Lifecycle Phases** (when we run side-effects).

The flat schema should be ordered exactly as the tool evaluates it.

```jsonc
{
  // --- 1. IDENTITY & CONTEXT ---
  // Global settings and variables available throughout the entire operation.
  "name": "", 
  "time-zone": "UTC", 
  "custom-string-patterns": {}, 

  // --- 2. EXECUTION STRATEGY ---
  // Defines how the release is delivered.
  "mode": "review", // or "auto"
  "review": { 
    // Settings strictly for PRs 
    "pull-request": { /* title-template, labels, etc. */ } 
  },
  "auto": { 
    // Settings strictly for direct execution
    "auto-strategy": { /* commit-types, etc. */ } 
  },

  // --- 3. SIDE EFFECTS & OVERRIDES ---
  // Scripts that run around the core logic, and dynamic config overrides.
  "command-hooks": {
    "base": { "pre": [], "post": [] },    // The global wrapper
    "prepare": { "pre": [], "post": [] }, // Runs before code is committed
    "publish": { "pre": [], "post": [] }  // Runs before tag/release is created
  },
  "runtime-config-override": { /* path, format */ }, // Applied AFTER base pre-hooks run

  // --- 4. THE VERSIONING ENGINE ---
  // How the tool calculates the next version.
  "initial-version": "0.1.0", 
  "version-files": [], 
  "commit-types": [], 
  "stop-resolving-commit-at": "", 
  "allowed-release-as-commit-types": "<ALL>", 
  "bump-strategy": { /* major, minor, patch, etc. */ }, 

  // --- 5. THE OUTPUT ARTIFACTS (Chronological) ---
  // The domain artifacts generated by the tool.
  "changelog": { /* path, templates, etc. */ }, 
  "commit": { /* header-template, local-files-to-commit, etc. */ }, 
  "tag": { /* create, tag-name-template, etc. */ }, 
  "release": { /* create-release-note, assets, etc. */ } 
}
```

## 2. The Orchestrator Refactor (`run.ts`)

Currently, `run.ts` runs base pre-hooks early, calculates context, and has an inline `if (runCtx.config.mode === "review")` in the middle of the flow.

To fix the variable export issue (e.g., exposing `zr-operation` before hooks run) and implement a clean Strategy Pattern, `run.ts` must be split into three strict stages: **Analyze**, **Hook**, and **Execute**. Using named exports strictly across these modules ensures the API remains clean and conventional for the backend.

### Stage 2.1: Analyze (No Side Effects)

Calculate what the application *intends* to do.

```typescript
import { determineOperationState } from "./tasks/operation.ts";

// 1. Setup & Resolve Config
const inputsResult = getInputsOrThrow(provider);
setupOperation(provider, inputsResult.inputs);
const configResult = await resolveConfigOrThrow(provider, inputsResult.inputs);

// 2. Determine State
// Inspects git context and config to determine "operation" and "jobs".
// Does NOT mutate anything or run commands.
const operationState = await determineOperationState(provider, configResult, inputsResult);
```

### Stage 2.2: Export & Base Hooks

Now that the state is known, export the variables and run the universal hooks.

```typescript
import { exportBaseOperationVariables } from "./tasks/export-variables.ts";
import { runCommandsOrThrow } from "./tasks/command.ts";

// 3. Export variables (Now `zr-operation` and `zr-jobs` are safely populated)
await exportBaseOperationVariables(provider, operationState);

// 4. Run Global Pre-Hooks
await runCommandsOrThrow(configResult.config.commandHooks.base, "pre");
```

### Stage 2.3: Delegate to Strategy

Remove the inline `if` statement. Map the mode string to a specific strategy function and execute it.

```typescript
import { executeReviewStrategy } from "./workflows/review.ts";
import { executeAutoStrategy } from "./workflows/auto.ts";
import type { StrategyFunction } from "./types/strategy.ts";

// Map available strategies
const strategies = {
  review: executeReviewStrategy,
  auto: executeAutoStrategy,
} satisfies Record<ConfigMode, StrategyFunction>;

// Execute the selected strategy
const executeStrategy = strategies[configResult.config.mode];
const finalRunCtx = await executeStrategy(provider, operationState);
```

## 3. Strategy Implementations (`review.ts` & `auto.ts`)

Both strategies will utilize shared domain utilities, but they orchestrate the lifecycle hooks differently.

### `executeReviewStrategy`

1. **Check PR State:** Does a PR exist?
2. **If NO PR (Prepare Flow):**
    * Calculate bump & changelog.
    * Run `command-hooks.prepare.pre`.
    * Commit changes to branch.
    * Create PR.
    * Run `command-hooks.prepare.post`.
3. **If PR MERGED (Publish Flow):**
    * Extract changelog/version.
    * Run `command-hooks.publish.pre`.
    * Create Tag & Release.
    * Run `command-hooks.publish.post`.

### `executeAutoStrategy`

For this initial refactoring branch, **do not implement the logic for `auto` mode.** 1. **Throw Error:**
    *Create the `executeAutoStrategy` function so the orchestrator (`run.ts`) can route to it.
    * Immediately `throw new Error("Auto execution mode is not yet implemented.")`.
    * The actual logic for direct commits and sequential lifecycle hooks will be built in the follow-up feature branch.

## 4. Variable Exports (`export-variables.md`)

Because you've shifted to the centralized `command-hooks` namespace, your documentation and internal logic for variable exports become much clearer.

* Variables previously under **Review mode - Propose** simply become **Prepare Phase Variables**. They are available when `command-hooks.prepare` runs, whether in `review` or `auto` mode.
* Variables previously under **Review mode - Release** become **Publish Phase Variables**. Available when `command-hooks.publish` runs.
